# Домашнее задание: «Микросервисы: принципы»  
## Задача 1: Выбор API Gateway

---

## Что такое API Gateway (простыми словами)

API Gateway — это как «единый вход» в нашу систему микросервисов. Он принимает запросы от клиентов, проверяет, кто отправил запрос, и перенаправляет его нужному сервису. Это помогает не «светить» внутреннюю архитектуру наружу и централизованно управлять безопасностью и маршрутизацией.

---

## Сравнительная таблица решений

4 популярных **открытых** решения, которые можно развернуть самостоятельно:

| Критерий | **Kong** | **NGINX + модули** | **Traefik** | **Apache APISIX** |
|----------|----------|-------------------|-------------|-------------------|
| **Маршрутизация** | По путям, доменам, заголовкам (через конфиг или API) | Гибкая, но требует ручного написания конфига | Автоматическая (особенно в Kubernetes) | Динамическая, без перезагрузки |
| **Аутентификация** | Плагины: JWT, OAuth2, Key-auth, LDAP | Только базовая, остальное — через Lua/модули | Через middleware, но меньше готовых плагинов | Много плагинов: JWT, OAuth2, HMAC |
| **HTTPS termination** | Встроенная поддержка SSL, можно автоматизировать через Let's Encrypt | Отличная поддержка, но настройка вручную | Автоматическое получение сертификатов | Поддержка SSL и автоматизация |
| **Сложность настройки** | Средняя (есть GUI в Enterprise) | Высокая (конфиги вручную) | Низкая (особенно в K8s) | Средняя (нужно знать etcd) |
| **Сообщество и документация** | Очень большое, много примеров | Огромное, но не всегда про API Gateway | Растёт, особенно в Cloud Native | Активное, но меньше материалов на русском |
| **Лицензия** | Apache 2.0 (Open Source) + Enterprise | BSD + коммерческие модули | Apache 2.0 | Apache 2.0 |

---

## Мой выбор: **Kong Gateway (Open Source)**

1. **Соответствует всем требованиям задания:**
   - Маршрутизация настраивается через декларативные конфиги или Admin API
   - Есть готовые плагины для проверки JWT, API-ключей, OAuth2
   - Поддерживает HTTPS termination «из коробки», можно подключить Let's Encrypt

2. **Хороший баланс для новичка:**
   - Много готовых примеров и туториалов
   - Можно начать с Docker-образа — не нужно сразу разбираться с компиляцией
   - Есть локальный режим (без базы данных) для тестов

3. **Перспективы роста:**
   - Если компания вырастет — можно перейти на Enterprise-версию с GUI и поддержкой
   - Интегрируется с Kubernetes, если позже будем мигрировать

4. **Безопасность:**
   - Регулярные обновления, активное сообщество
   - Поддержка mTLS, rate limiting, CORS — всё через плагины

## Задача 2: Выбор брокера сообщений

---

## Что такое брокер сообщений (простыми словами)

Брокер сообщений — это «почтовое отделение» для наших микросервисов. Сервисы не общаются друг с другом напрямую, а отправляют сообщения в брокер. Это позволяет сервисам не зависеть от доступности друг друга (асинхронность) и легче масштабироваться.

---

## Сравнительная таблица решений

Я выбрал 3 популярных решения с открытым исходным кодом, которые часто используют в индустрии:

| Критерий | **Apache Kafka** | **RabbitMQ** | **ActiveMQ Artemis** |
|----------|------------------|--------------|----------------------|
| **Кластеризация** | Нативная, высокая надёжность (KRaft/ZooKeeper) | Есть (Quorum Queues), но сложнее в настройке | Поддерживается, но требует тонкой настройки |
| **Хранение на диске** | Логи хранятся на диске, можно перечитывать | Сообщения сохраняются, но ориентировано на доставку | Полная поддержка персистентности |
| **Скорость работы** | Очень высокая (тысячи сообщений в секунду) | Высокая (но ниже Kafka при больших нагрузках) | Средняя/Высокая |
| **Форматы сообщений** | Байты (любой формат: JSON, Avro, Protobuf) | Байты (любой формат) | Байты (любой формат) |
| **Разделение прав (ACL)** | Поддержка ACL для топиков | Гибкая система прав (users, vhosts, tags) | Поддержка ACL |
| **Простота эксплуатации** | Сложно (требует мониторинга, tuning) | Проще (есть удобный UI, понятная логика) | Средне (много конфигурации) |
| **Модель обмена** | Log-based (поток событий) | Queue/Exchange (задачи, RPC) | Queue/Topic (гибрид) |

---

## Мой выбор: **RabbitMQ**

1. **Соответствует требованиям задания:**
   - **Кластеризация:** Поддерживает кластеры и кворумные очереди для надёжности.
   - **Диск:** Сообщения сохраняются на диск, не теряются при перезагрузке.
   - **Скорость:** Достаточно высокая для старта и большинства задач бизнеса.
   - **Форматы:** Работает с любыми данными (для нас это будет JSON).
   - **Права доступа:** Есть встроенная система пользователей, виртуальных хостов и прав на очереди.
   - **Эксплуатация:** Есть удобный веб-интерфейс для мониторинга очередей и управления.
